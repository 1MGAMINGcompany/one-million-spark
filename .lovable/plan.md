# Plan: Move native parimutuel events to a true pool model\n\n## Why\n\nYou're right — under a parimutuel pool, the platform never fronts payouts. Losers fund winners. The relayer should only ever need **MATIC for gas**, not USDC.e for liability.\n\nToday, the code partially honors this for fees (transferFrom → Treasury wallet ✅) but **does not pool the stakes**:\n- The 95% stake is never moved on-chain at submit time. The user's USDC.e just sits in their own Privy wallet, tracked only in DB (`prediction_entries.pool_usd`).\n- At claim time, `transferUsdcFromTreasury()` in `prediction-claim` pays the winner from the **relayer wallet** — which has $0 USDC.e. That's why I (incorrectly) flagged it as a blocker earlier.\n\nThis plan fixes the model so it matches your description.\n\n## The corrected money flow\n\n| Step | Action | Token | Funded by |\n|---|---|---|---|\n| 1 | User on-ramps via Privy | USDC.e → user's wallet | User |\n| 2 | Submit: `transferFrom` 5% fee → **Treasury** | USDC.e | User |\n| 3 | Submit: `transferFrom` 95% stake → **Pool wallet (per-event or global)** | USDC.e | User |\n| 4 | All gas (steps 2 + 3 + payouts) | MATIC | Relayer |\n| 5 | Claim: `transfer` reward → winner from **Pool wallet** | USDC.e | The pool itself (losers + winners' stakes) |\n\n## Changes\n\n### 1.
